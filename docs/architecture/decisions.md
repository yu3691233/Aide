## ADR-001: 选择 Flask 作为 Web 框架

### 状态

已采纳

### 背景

需要一个轻量级的 Web 框架来提供 REST API 服务。

### 决策

选择 Flask 作为 Web 框架。

### 原因

1. **简单轻量**：Flask 是 Python 最轻量的 Web 框架，学习曲线低
2. **快速开发**：适合快速原型开发，减少 boilerplate 代码
3. **扩展性强**：通过 Blueprint 支持模块化，便于后期重构
4. **社区活跃**：丰富的扩展和文档，问题容易找到解决方案
5. **足够使用**：对于本地网络应用，性能完全足够

### 后果

**优点**：
- 开发速度快
- 代码简洁易读
- 易于维护和扩展

**缺点**：
- 性能不如 FastAPI
- 异步支持较弱
- 需要手动实现一些功能

### 替代方案

- **FastAPI**：性能更好，异步支持强，但学习曲线较陡
- **Django**：功能全面，但过于重量级
- **Tornado**：异步支持好，但社区较小

## ADR-002: 使用 ADB 进行设备通信

### 状态

已采纳

### 背景

需要一种方式让 Android 设备和 PC 进行通信。

### 决策

使用 ADB (Android Debug Bridge) 进行设备通信。

### 原因

1. **原生支持**：Android 原生调试工具，无需额外安装
2. **功能丰富**：支持文件传输、命令执行、端口转发
3. **稳定可靠**：经过多年验证，稳定性高
4. **无线支持**：支持无线 ADB 连接，方便使用
5. **无需额外依赖**：Android 设备自带，无需安装额外应用

### 后果

**优点**：
- 无需额外安装应用
- 功能强大，支持多种操作
- 稳定可靠

**缺点**：
- 需要开启开发者选项
- 无线连接不如 WebSocket 稳定
- 需要处理连接断开重连

### 替代方案

- **WebSocket**：实时性好，但需要 Android 端运行服务
- **HTTP 长轮询**：实现简单，但实时性差
- **蓝牙**：无需网络，但速度慢，配对麻烦

## ADR-003: 使用 Kotlin + Compose 开发 Android 应用

### 状态

已采纳

### 背景

需要开发 Android 客户端应用。

### 决策

使用 Kotlin + Jetpack Compose 开发 Android 应用。

### 原因

1. **现代语言**：Kotlin 是 Android 官方推荐语言，语法简洁
2. **声明式 UI**：Compose 简化 UI 开发，代码更直观
3. **性能优秀**：编译为原生代码，性能接近 Java
4. **类型安全**：编译时类型检查，减少运行时错误
5. **协程支持**：原生支持协程，异步编程更简单

### 后果

**优点**：
- 开发效率高
- 代码简洁易读
- 性能优秀
- 类型安全

**缺点**：
- 学习曲线较陡（相比 Java）
- 生态不如 Java 成熟
- 部分库需要适配

### 替代方案

- **Java + XML**：传统方案，生态成熟，但代码冗余
- **Flutter**：跨平台方案，但性能不如原生
- **React Native**：跨平台方案，但需要 JavaScript 知识

## ADR-005: 使用 JSON 文件存储状态

### 状态

已采纳

### 背景

需要一种方式存储应用状态（设置、任务、历史等）。

### 决策

使用 JSON 文件存储状态。

### 原因

1. **简单易用**：JSON 格式简单，易于读写
2. **无需依赖**：Python 原生支持，无需额外安装数据库
3. **便于调试**：可以直接查看和编辑文件
4. **适合小数据量**：对于个人项目，数据量不大
5. **版本控制友好**：可以纳入版本控制

### 后果

**优点**：
- 实现简单
- 无需额外依赖
- 便于调试
- 版本控制友好

**缺点**：
- 并发写入需要加锁
- 查询性能不如数据库
- 数据量大时性能下降

### 改进措施

- 使用原子写入（临时文件 + os.replace）
- 使用线程锁保护并发写入
- 定期备份重要数据

### 替代方案

- **SQLite**：轻量级数据库，支持复杂查询，但需要额外依赖
- **PostgreSQL**：功能强大，但过于重量级
- **Redis**：高性能缓存，但需要额外服务

## ADR-006: 使用 Blueprint 组织路由

### 状态

已采纳

### 背景

随着功能增加，`phone_chat_bridge.py` 文件越来越大，难以维护。

### 决策

使用 Flask Blueprint 将路由拆分到独立模块。

### 原因

1. **模块化**：每个功能模块独立，职责单一
2. **可维护性**：代码分散到多个文件，易于维护
3. **可测试性**：每个模块可以独立测试
4. **团队协作**：不同开发者可以并行开发不同模块
5. **代码复用**：模块可以在不同项目中复用

### 后果

**优点**：
- 代码结构清晰
- 易于维护和扩展
- 便于团队协作
- 便于测试

**缺点**：
- 需要管理模块间的依赖
- 需要处理模块初始化顺序
- 增加了代码复杂度

### 模块划分

- `message_routes.py`：消息桥接
- `screenshot_routes.py`：截图功能
- `ide_routes.py`：IDE 控制
- `settings_routes.py`：设置管理
- `xiaomengling_routes.py`：Aide AI
- `device_routes.py`：设备管理
- `frp_routes.py`：FRP 管理
- `app_routes.py`：应用管理
- `event_routes.py`：事件流

## ADR-007: 使用环境变量管理敏感信息

### 状态

已采纳

### 背景

API Key 等敏感信息不能硬编码在代码中。

### 决策

使用环境变量管理敏感信息，通过 `.env` 文件配置。

### 原因

1. **安全性**：敏感信息不会泄露到版本控制
2. **灵活性**：不同环境可以使用不同配置
3. **标准化**：Python 社区的标准做法
4. **易于管理**：通过 `.env` 文件集中管理
5. **工具支持**：`python-dotenv` 库支持自动加载

### 后果

**优点**：
- 安全性高
- 配置灵活
- 易于管理
- 符合最佳实践

**缺点**：
- 需要额外配置
- 部署时需要设置环境变量
- 调试时可能不方便

### 实现方式

```python
# .env 文件
MINIMAX_API_KEY=your_api_key_here
BRIDGE_DIR=F:\AideLink\server

# Python 代码
import os
from dotenv import load_dotenv

load_dotenv()
api_key = os.getenv('MINIMAX_API_KEY')
```

## ADR-008: 使用原子写入保护 JSON 文件

### 状态

已采纳

### 背景

多线程环境下，JSON 文件可能被损坏。

### 决策

使用原子写入（临时文件 + os.replace）保护 JSON 文件。

### 原因

1. **数据安全**：避免写入过程中断导致文件损坏
2. **并发安全**：配合线程锁，支持多线程并发写入
3. **简单有效**：实现简单，效果明显
4. **跨平台**：os.replace 在所有平台都是原子操作
5. **无需额外依赖**：Python 原生支持

### 后果

**优点**：
- 数据安全性高
- 并发安全
- 实现简单
- 无需额外依赖

**缺点**：
- 需要额外的磁盘空间（临时文件）
- 写入速度略慢
- 需要处理临时文件清理

### 实现方式

```python
import json
import os
import tempfile
from threading import Lock

_file_locks = {}

def atomic_write_json(filepath: str, data: dict):
    lock = _file_locks.setdefault(filepath, Lock())
    
    with lock:
        # 写入临时文件
        dir_name = os.path.dirname(filepath)
        with tempfile.NamedTemporaryFile(
            mode='w',
            dir=dir_name,
            delete=False,
            suffix='.tmp'
        ) as f:
            json.dump(data, f, ensure_ascii=False, indent=2)
            temp_path = f.name
        
        # 原子替换
        os.replace(temp_path, filepath)
```

## ADR-009: 使用 SSE 实现实时事件推送

### 状态

已采纳

### 背景

需要将服务端事件实时推送到客户端。

### 决策

使用 Server-Sent Events (SSE) 实现实时事件推送。

### 原因

1. **简单易用**：基于 HTTP，实现简单
2. **浏览器支持**：现代浏览器原生支持
3. **单向通信**：适合服务端向客户端推送事件
4. **自动重连**：浏览器自动处理断线重连
5. **无需额外依赖**：基于标准 HTTP 协议

### 后果

**优点**：
- 实现简单
- 浏览器支持好
- 自动重连
- 无需额外依赖

**缺点**：
- 单向通信（服务端 → 客户端）
- 不支持二进制数据
- 连接数有限制

### 替代方案

- **WebSocket**：双向通信，但实现复杂
- **长轮询**：实现简单，但实时性差
- **gRPC**：高性能，但学习曲线陡

## ADR-010: 使用 Hilt 进行依赖注入

### 状态

已采纳

### 背景

Android 应用中需要管理依赖关系。

### 决策

使用 Hilt 进行依赖注入。

### 原因

1. **官方推荐**：Google 官方推荐的依赖注入框架
2. **编译时检查**：编译时验证依赖关系，减少运行时错误
3. **与 Jetpack 集成**：与 ViewModel、Navigation 等 Jetpack 组件深度集成
4. **简化代码**：自动生成样板代码，减少手动配置
5. **易于测试**：支持 Mock 依赖，便于单元测试

### 后果

**优点**：
- 类型安全
- 编译时检查
- 简化代码
- 易于测试

**缺点**：
- 学习曲线较陡
- 编译时间增加
- 需要 KSP/kapt 支持

### 替代方案

- **Koin**：轻量级，学习曲线低，但运行时检查
- **Dagger**：功能强大，但配置复杂
- **手动注入**：简单直接，但代码冗余
