# AideLink 桌面浮窗与开发随记升级任务包

- 状态：待继续讨论 / 待选择 IDE / 未派发
- 日期：2026-07-17
- 建议项目路径：`docs/AIDELINK_DESKTOP_CAPTURE_FLOATING_WINDOW_TASKS.md`
- 目的：把本轮产品讨论固化为可持续补充、可拆分派发、不会随聊天上下文丢失的任务包。

## 1. 背景问题

多 Agent 工作时，用户不应只是等待。用户会继续测试、观察和思考，并不断产生：

- 当前实现中的问题；
- 界面细节和组件错误；
- 当前任务的后续修改；
- 新功能想法；
- 以后再做的优化方向；
- 尚未整理完整的短句和疑问。

目前这些内容容易停留在聊天上下文、临时记事或用户记忆中。模型会话上下文有限，讨论较久但尚未执行时，已确认的细节可能在换会话、换 IDE 或延迟派发后丢失。

本功能要把“讨论和发现问题”本身变成 AideLink 可保存、可整理、可转换、可派发的正式工作流。

## 2. 已确认的产品决定

1. 桌面浮窗的第一用途不是缩小版 Web 管理页，而是多 Agent 工作期间的开发现场捕获入口。
2. 原 APP 端“灵感”功能升级为统一的“开发随记”，不再新建一套平行数据。
3. 开发随记允许只写几个字，不要求立即形成完整任务。
4. 原始随记必须独立保存，不能依赖当前聊天会话继续存在。
5. Web 端已有“组件定位 + 智能提示词”半成品，第一阶段先接入浮窗，不从零重写。
6. 组件定位信息必须结构化保存，不能只把“这个下拉栏”拼成一句文字。
7. 用户可以选择一个指定 IDE，将一条或多条随记整理成完整任务草稿。
8. IDE 整理后先返回任务预览，用户确认后才创建任务或派发。
9. 当前 Agent 完成后，用户决定随记是：
   - 继续当前任务；
   - 创建当前任务的后续或子任务；
   - 创建独立新任务；
   - 保留待整理；
   - 合并、归档或忽略。
10. 原始随记、IDE 整理结果和最终任务之间必须保留关联，避免整理后丢失原始现场信息。
11. 托盘左键最终改为显示或隐藏浮窗；完整 Web 页面保留为二级入口。
12. 第一版优先打通“快速记录 → 组件关联 → IDE 整理 → 用户决策”，完整任务列表能力可随后逐步补齐。

## 3. 产品定位

> Agent 工作时，人继续思考；AideLink 负责接住开发现场里还没来得及处理的内容。

开发随记不是普通备忘录，而是带有项目、任务、Agent、界面组件、截图和来源环境等上下文的“任务素材”。

## 4. 现有能力与需要复用的部分

开发前必须先只读核对实际代码，确认以下现有实现的位置、接口和完成度：

- APP 端“灵感”数据模型、接口和页面；
- Web 端任务列表及任务增删改功能；
- 浏览器插件组件定位半成品；
- 智能提示词生成流程；
- 当前项目、当前任务和 IDE 绑定信息；
- 托盘点击打开 Web 的现有实现；
- Windows 窗口控制和置顶能力；
- MCP、OpenCode、Codex 深链、窗口注入、手动技能等 IDE 交互通道；
- 任务创建、派发、`result_ref`、`pending_test` 和验证流程。

原则：优先扩展现有数据和接口，不复制一套桌面专用任务系统。

## 5. 外部方案调研结论

### 5.1 可参考

- **Raycast Notes**
  - 值得借鉴：全局快捷键、浮动窗口、不中断当前工作流的快速记录。
  - 不需要照搬：完整笔记系统和富文本编辑器。

- **Marker.io**
  - 值得借鉴：在问题发生的位置直接记录、自动附带截图和技术元数据、将简短反馈整理成可执行问题。
  - 不需要照搬：团队 SaaS、外部项目管理平台和完整会话回放体系。

- **rrweb**
  - 值得以后评估：页面操作回放、控制台记录、错误前后行为证据。
  - 第一版不引入：体积、数据量、隐私和存储复杂度都超出当前最小需求。

### 5.2 当前采用判断

- 直接复用 AideLink 已有 APP 灵感、浏览器插件、提示词和任务接口。
- 浮窗只做新界面层和捕获工作流，不新建第二套后端。
- 组件定位不能依赖单个 CSS 选择器或框架专用定位方式，应保存多层身份和截图兜底。
- 会话回放、网络日志和控制台日志放入后续增强，不阻塞第一版。

## 6. 核心数据建议

现有“灵感”实体优先扩展，而不是立即改名或迁移全部历史数据。

建议增加或映射以下字段：

```text
Capture / Inspiration
- id
- project_id
- note
- type: idea | bug | optimization | question | other
- status: inbox | organizing | organized | converted | archived
- related_task_id: optional
- related_run_id: optional
- source: android | desktop_float | browser_plugin | web
- context_refs[]
- organized_versions[]
- converted_task_ids[]
- created_at
- updated_at
```

### ContextReference

```text
ContextReference
- id
- type: web_component | android_component | screenshot | page | file | task | run | clipboard
- source
- label
- payload
- created_at
```

### WebComponentReference 最小字段

```text
- page_title
- page_url
- component_type / role
- visible_text
- accessible_name
- nearby_text
- stable_attributes: id | name | data-testid | aria-label
- selector_candidates[]
- dom_path_summary
- bounding_rect
- viewport
- screenshot_ref
- captured_at
```

要求：

- 不把完整 DOM 永久硬塞进任务文本；
- 保存结构化引用，整理任务时按需解析；
- 允许定位失败时依靠可见文字、附近结构、坐标和截图兜底；
- 敏感字段、输入内容和截图需要后续制定脱敏规则。

## 7. 目标工作流

```text
Agent 开始执行任务
    ↓
用户继续测试、观察和思考
    ↓
浮窗快速记录短句
    ↓
可选关联当前任务 / Agent Run / 页面组件 / 截图
    ↓
随记独立保存，不依赖聊天会话
    ↓
用户选择一条或多条随记
    ↓
选择指定 IDE“整理为任务”
    ↓
IDE 返回结构化任务草稿
    ↓
用户预览并决定：
继续当前任务 / 后续子任务 / 独立任务 / 暂存
    ↓
确认后才创建或派发
```

## 8. 任务拆解

以下任务均为“待派发”，当前不自动创建 AideLink 运行时任务，也不指定执行 IDE。

---

### F0：现状审计与最小方案冻结

**目标**

只读确认现有 APP 灵感、Web 插件组件定位、智能提示词、托盘和任务 API 的真实完成度，形成最小改造方案。

**产出**

- 现有代码路径清单；
- 可直接复用接口；
- 重复数据和潜在冲突；
- 第一版最小范围；
- 修改文件范围建议；
- 外部依赖及许可证结论。

**验收**

- 不凭文档猜测代码；
- 明确“复用 / 修改适配 / 参考 / 自主实现”；
- 不修改任何业务代码。

**适合**

任何额度较低但可进行只读分析的 IDE。该任务不值得消耗大量 Codex 额度。

---

### F1：开发随记数据模型升级

**目标**

在兼容 APP 端现有灵感数据的前提下，支持来源、类型、状态、任务关联、Agent Run 关联、上下文引用和任务转换记录。

**重点**

- 优先向后兼容；
- 避免一次性破坏性迁移；
- 原始文字永久保留；
- IDE 整理结果不得覆盖原文；
- 一条随记可产生多个整理版本或关联任务。

**验收**

- APP 旧灵感仍可读取；
- 新字段缺失时有稳定默认值；
- 数据迁移可回退或重复执行；
- 有模型和 API 测试。

**建议分配**

架构和迁移较关键，适合能力较强且善于跨模块验证的 IDE。Codex 额度有限时，可把实现交给 Trae，最后由 Codex 只做审查。

---

### F2：统一开发随记 API

**目标**

提供桌面浮窗、APP、Web 和浏览器插件共同使用的接口。

**最小接口**

- 创建随记；
- 查询当前项目随记；
- 修改文字、类型和关联；
- 批量选择；
- 归档；
- 添加或删除 `context_ref`；
- 标记正在整理；
- 保存 IDE 返回的任务草稿；
- 转换为正式任务。

**验收**

- 多端不需要各自维护业务规则；
- 重复请求具有基本幂等保护；
- 整理失败后随记不会丢失或卡死；
- 有权限、空值和非法状态测试。

---

### F3：浏览器插件组件引用标准化

**目标**

把现有半成品的组件定位结果转换为统一 `WebComponentReference`，并能发送到桌面浮窗或开发随记 API。

**第一版范围**

- 保留现有选中组件交互；
- 输出页面、组件、文字、稳定属性、候选选择器、坐标和局部截图；
- 浮窗显示“组件胶囊”；
- 用户输入短句后一起保存；
- 不要求第一版直接定位到源码文件。

**必须避免**

- 只保存 CSS selector；
- 只保存坐标；
- 直接把整个 DOM 发送给模型；
- 因 React、Vue 或普通 HTML 不同而建立三套数据结构。

**验收**

至少验证：

- 普通 HTML 控件；
- React/Vue 渲染控件；
- 多个相邻下拉栏；
- 动态列表；
- iframe 或 Shadow DOM 无法定位时给出明确降级；
- 页面更新后仍能通过多层信息识别目标。

**建议分配**

可先让熟悉前端和浏览器插件的 IDE完成。Codex 只保留协议审查和边界测试。

---

### F4：桌面浮窗外壳

**目标**

建立桌面端浮窗容器，不先塞入全部 Web 功能。

**功能**

- 托盘左键显示 / 隐藏；
- 浮窗已被遮挡时置前；
- 可固定最前；
- 可拖动和调整大小；
- 记住位置、大小和置顶状态；
- 提供打开完整 Web 页面入口；
- 可选全局快捷键；
- 服务异常时显示明确状态。

**验收**

- 不重复启动多个浮窗；
- 多显示器、DPI 缩放下位置合理；
- 浮窗关闭不退出服务；
- 托盘右键仍能退出和打开完整页面；
- 不影响现有 Android、Web 和 MCP 服务。

**建议分配**

边界清晰，适合 Trae 或其他额度充足的 IDE。先单独完成窗口生命周期，不与业务 UI 混改。

---

### F5：浮窗“开发随记”首页

**目标**

实现第一版核心交互。

**布局优先级**

1. 当前项目和当前任务关联；
2. 快速输入框；
3. 已选组件胶囊；
4. 类型快捷标签；
5. 最近开发随记；
6. Agent 工作状态和待处理提醒；
7. 进入完整任务列表的入口。

**交互**

- `Enter` 保存，`Shift + Enter` 换行；
- 保存后不打断 Agent；
- 自动关联当前项目；
- 可选自动关联当前任务或当前 Run；
- 允许一键解除关联；
- 最近随记支持编辑、归档、整理为任务；
- 无网络或服务异常时不能静默丢内容。

**验收**

- 用户可在几秒内完成记录；
- 无需填写标题、优先级或执行者；
- 浮窗尺寸有限时不拥挤；
- 键盘操作完整；
- 有空状态、加载状态和错误状态。

---

### F6：指定 IDE“整理为任务”

**目标**

将一条或多条开发随记及其上下文交给用户指定的 IDE，生成完整任务草稿。

**输入包**

- 原始随记；
- 当前项目；
- 当前任务和 Run；
- 组件引用；
- 截图；
- 用户选定的任务关系倾向；
- AideLink 的任务约束和验证规则。

**输出草稿**

- 标题；
- 背景；
- 目标；
- 精确位置；
- 复现步骤；
- 当前结果；
- 预期结果；
- 修改范围；
- 非目标；
- 验收标准；
- 风险和待确认问题；
- 建议继续当前任务还是新建任务。

**实现要求**

- 适配现有 IDE 通道，不建立新的模型调用中心；
- 支持 MCP、API、深链、窗口注入或手动复制等现有能力；
- IDE 只负责整理，不得自动派发或修改代码；
- 保存原始输入与返回版本；
- 超时、失败或返回格式错误时可重试或切换 IDE。

**建议分配**

这是 AideLink 的核心协议任务。Codex 额度少时，先由其他 IDE 实现通用适配器和 UI，Codex 最后只检查任务包协议、状态和失败恢复。

---

### F7：任务预览与转换决策

**目标**

让用户在任务草稿生成后明确决定下一步。

**可选动作**

- 继续当前任务；
- 作为当前任务后续或子任务；
- 创建独立任务；
- 合并到已有任务；
- 返回重新整理；
- 保留待整理；
- 归档。

**规则**

- 不自动派发；
- 不偷偷改变已进入 `pending_test` 的任务目标；
- “继续当前任务”应创建新的执行轮次或显式重新打开；
- 转换后仍能回看原始随记、组件证据和整理版本；
- 转换成功后写入关联任务 ID。

**验收**

- 重复点击不会重复创建任务；
- 创建失败时随记仍保留；
- 用户可修改草稿后再转换；
- 任务状态符合现有运行时守卫。

---

### F8：Agent 完成后的随记整理提醒

**目标**

当某个 Agent Run 完成时，汇总该执行期间产生并关联的开发随记。

**表现**

- 显示本轮新增随记数量；
- 按问题、优化、新功能和疑问分组；
- 推荐“继续当前任务”或“创建新任务”，但不自动执行；
- 支持批量选择和批量整理；
- 没有随记时不增加额外打扰。

**验收**

- 只汇总正确时间范围和关联关系；
- 同一随记不重复提醒；
- 任务失败、完成和待测试状态均有合理入口。

---

### F9：完整任务列表逐步接入浮窗

**目标**

在核心捕获链路稳定后，再补齐桌面浮窗中的任务管理能力。

**后续范围**

- 查看和筛选任务；
- 新增、编辑、删除；
- 状态操作；
- 查看执行者和结果；
- 待测试和需关注队列；
- 跳转完整 Web 页面。

**非第一阶段阻塞项**

不要求在 F4/F5 完成前实现全部 Web/App 功能。

---

### F10：测试、隐私与迁移验收

**必须覆盖**

- APP 旧灵感兼容；
- 桌面、Web、APP 同步；
- 多端同时编辑；
- 服务中断后的本地草稿保护；
- 敏感输入和截图处理；
- 插件权限和 CSP 限制；
- 动态页面、iframe、Shadow DOM 降级；
- 多显示器和 DPI；
- IDE 整理失败、超时、返回非结构化文本；
- 任务转换幂等；
- 当前任务完成后再继续的状态正确性。

## 10. 低 Codex 额度下的分配建议

### 可优先交给 Trae 或其他额度充足 IDE

- F0 只读审计；
- F3 浏览器插件适配；
- F4 浮窗外壳；
- F5 UI；
- F9 任务列表界面；
- 常规单元测试和文档。

### 建议保留给 Codex 审查或最后处理

- F1 数据迁移和兼容边界；
- F2 状态与幂等；
- F6 IDE 整理协议；
- F7 与现有任务状态机的衔接；
- 跨模块最终验收。

### MiniMax Code 可承担

- 边界清楚、路径明确的 UI 或文档任务；
- 不依赖第三方 MCP 的独立修改；
- 已有清晰验收步骤的小型测试修复。

## 11. 暂不做

- 完整复制 Marker.io、Raycast 或其他产品；
- 新建独立桌面任务后端；
- 自动把每条随记变成任务；
- 自动把任务派给 Agent；
- 第一版引入完整会话回放；
- 第一版记录全部网络和控制台日志；
- 仅依赖一个 CSS selector 定位组件；
- 在浮窗里重建完整 IDE、聊天或文件管理器；
- 为了视觉效果先做复杂动画和看板拖拽。

## 12. 后续仍需讨论的问题

1. UI 名称继续叫“灵感”，还是升级为“开发随记”并保留灵感分类？
2. 浮窗默认宽高、收缩态和边缘吸附是否进入第一版？
3. 默认自动关联当前任务，还是只关联当前项目？
4. 如何准确判断当前用户正在关注哪个任务或 IDE？
5. 多条随记整理时，默认合成一个任务还是先由 IDE 判断？
6. 组件截图保存多久，是否默认本地存储？
7. Web 组件引用是否需要记录控制台错误和最近一次失败请求？
8. IDE 整理任务使用统一 JSON 协议，还是先兼容纯文本返回？
9. 继续当前任务时，采用新 Run、重新打开任务还是后续子任务？
10. 浮窗第一版是否同时支持查看完整任务列表，还是只保留入口？

## 13. 第一阶段最小成功标准

用户可以在 Agent 工作期间：

1. 点击托盘打开浮窗；
2. 输入几个字保存开发随记；
3. 可从现有浏览器插件附加一个已选 Web 组件；
4. 随记独立保存，不因聊天结束而丢失；
5. 选择一个 IDE 将随记整理成任务草稿；
6. 查看并修改预览；
7. 决定继续当前任务或创建新任务；
8. 确认后进入现有任务系统；
9. 原始随记、组件证据和最终任务可互相追溯。

达到以上链路后，再继续扩大浮窗的完整任务管理功能。
