# AideLink 产品原则

> 这是 AideLink 的产品边界说明，不是功能清单，也不是技术债记录。
>
> 目的：在后续开发里持续提醒自己，AideLink 要做的是“辅助用户开发其他项目”的差异化工作流，而不是复刻已有 IDE 的默认范式。

## 1. 产品定位

AideLink 不是另一个 ChatGPT，也不是另一个 IDE。

它的核心是一个面向开发协作的前置调度层：

- 先记录灵感、待办、问题和不成熟的想法。
- 再由用户决定何时派发、派发给哪个 IDE。
- 再在执行过程中隔离上下文，避免不同任务互相污染。
- 再根据 IDE 的能力、额度和当前状态做分流决策。

用户熟悉的 IDE 交互方式，只是上手入口，不是产品终点。

## 2. 为什么要有任务列表

任务列表的意义，不是“新建后立刻开跑”，而是“先收纳，再判断”。

它解决的是开发过程中很真实的三类场景：

- 临时冒出来的灵感，先记下来。
- 方案还不成熟，先放在待派发里。
- 当前 IDE 正在工作，别把其他任务混进上下文。

因此，任务列表本质上是一个“任务缓冲区”和“派发决策池”。

## 3. 任务流边界

### 3.1 应该进入任务列表的内容

- 已经明确是开发任务、修复任务、验证任务或需要后续执行的工作项。
- 用户希望先保存、稍后再派发的想法。
- 需要分发给某个 IDE 处理，但暂时不应该马上占用该 IDE 的上下文。

### 3.2 不应该进入任务列表的内容

- 普通聊天消息。
- 快捷回复。
- 仅用于当前对话的补充说明。
- 已经在执行中的任务线程里的普通对话。

这类内容应该留在聊天上下文里，不污染任务列表。

## 4. 派发原则

任务只有在“派发”时，才真正进入 IDE 执行上下文。

派发前：

- 任务只是候选项。
- 用户可以继续整理、合并、拆分、延后。
- 不应消耗 IDE 额度。

派发后：

- 任务进入指定 IDE 的工作上下文。
- 任务状态才算真正开始。
- 当前上下文应尽量保持纯净，只包含这个任务相关内容。

## 5. 反馈原则

任务反馈不是新任务。

反馈的正确归属方式是：

- 绑定到原任务。
- 作为原任务线程里的追加信息。
- 作为同一任务的后续聊天记录，而不是新的待派发条目。

这样做的目的，是保持任务生命周期完整，避免反馈重新污染任务列表。

## 6. IDE 选择原则

AideLink 要支持用户按 IDE 能力做判断，而不是默认把所有事情都塞进同一个 IDE。

这意味着：

- 可以先把任务存起来。
- 可以等当前 IDE 空闲后再派发。
- 可以根据任务复杂度决定是否使用能力一般但更便宜的 IDE。
- 可以保留能力更强的 IDE 给更复杂的任务。

这也是 AideLink 和常见“一个输入框直接进一个模型/一个 IDE”的不同点。

## 7. 与成熟方案的关系

AideLink 不追求把别人已经做得很成熟的功能重新做一遍。

要优先做的是：

- 别人没有的工作流。
- 适合开发协作的上下文隔离。
- 任务收纳、派发、反馈、执行分离。
- 能让用户更快上手，但最终仍然保留自己的工作方式。

如果某个功能只是“看起来像某个成熟产品”，但没有形成 AideLink 的独特协作价值，就不应该成为默认方向。

## 8. 改动时的自检清单

每次改任务、聊天、反馈、派发相关代码前，先检查这几个问题：

- 这个内容是在“收纳”还是在“执行”？
- 它应该进入任务列表，还是只留在聊天里？
- 它会不会污染当前任务上下文？
- 它会不会让不同 IDE 的任务混在一起？
- 它是在增强 AideLink 的差异化，还是在复刻现成方案？

只要回答不清楚，就先停一下，不要急着把它做成默认行为。

## 9. 提交原则

为了避免工作区里长期堆积已完成改动，提交时遵守下面的规则：

- 已完成功能要尽快提交，不要长期丢在工作区。
- 只有明显未完成、实验性验证、一次性日志、截图或临时产物，才允许暂时保留。
- 如果一个改动已经完成了主要功能验证，就应当作为一个独立提交落盘。
- 提交时优先保持语义完整，不要把完成的功能和未完成的试验混在一起。

这个原则的目标不是“追求干净的 git 状态”，而是保证每个已完成结果都能被明确保存、回溯和复用。
